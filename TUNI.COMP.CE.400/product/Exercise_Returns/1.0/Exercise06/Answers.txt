4. How was the computation time/delay added to the simulation on both ARM simulation and HW-accelerated simulation? What function/s, variable/s, and calculations are used?
Delays are added using delay_ns() function in sc_kvazaar.c.
oIntra rough search delay: search_intra_rough_delay() adds delay based on width, delay_c, and search_intra_rough_percentage_c.
oRest of the encoder delay: search_delay() adds delay based on cu_width, delay_c, and rest_percentage_c.
These delays simulate processing times for each part of the encoder.

5. Do you think the computation time/delay is calculated accurately enough?
The calculations are based on profiling data (from gprof), but might not be 100% accurate in real-world hardware due to factors like unaccounted bottlenecks.

9. What does using wait() in delay_ns() cause?
Using wait() in delay_ns() makes the simulation timed, so delays consume simulation time, making the encoding time longer. Without it, the simulation is untimed and runs faster, i.e., the encoding time was shorter.

10. By looking at the two different waveforms, what kind of problems not using wait() causes?
I can see signal overlaps and fast switching in the waveform when not using wait(). Using wait() correctly spaces out signal transitions, ensuring accurate timing and proper simulation behavior.
